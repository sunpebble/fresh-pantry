# FreshPantry 多语言支持设计（zh-Hans / en / ja / fr）

日期：2026-07-05
状态：已确认

## 目标

FreshPantry 完整支持简体中文、英语、日语、法语四种语言，覆盖 UI 文案、报错信息、
通知、系统权限描述、以及全部内置食谱内容。语言跟随系统设置（用户可在
iOS 设置 → App 中单独切换），不做 app 内语言选择器。译文由 AI 产出直接使用。

## 现状（2026-07-05 调研）

- iOS app 开发语言为 zh-Hans，约 90–100 条硬编码中文字符串，零本地化基础设施
  （无 .xcstrings / Localizable.strings / `String(localized:)`）。
- API（Cloudflare Worker）有 6 条中文错误 message，iOS 端部分按状态码自行映射。
- 内置食谱 364 道（`apps/ios/FreshPantry/Resources/howtocook.json`，随 app 打包），
  菜名/描述/食材/单位/步骤/分类全部为中文，由 recipe-pipeline 产出。
- 日期格式化多处硬编码 `zh_CN` / `en_US_POSIX` locale。

## 设计

### 1. iOS 字符串目录（语义化 key 重构）

- 新建 String Catalog `Localizable.xcstrings`，源语言 zh-Hans，含 en / ja / fr 译文。
- 全部硬编码 UI 字符串（主 app ~193 处 `Text()`、Widget ~8 处、各 alert/toast）
  改为语义化 key，命名规范 `功能.组件.用途`，例如：
  - `tab.home`、`tab.inventory`、`tab.recipes`、`tab.shopping`、`tab.settings`
  - `settings.backup.exportFailed`、`recipe.expiringIn`、`paywall.title`
- 带插值的字符串使用 catalog 占位符（`%@` / `%lld`）。
- 数量类文案使用 xcstrings 复数变体（en / fr 需要单复数；zh / ja 单一形式）。
- Widget（FreshPantryWidgets）与 ShareExtension 两个 target 挂接字符串目录
  （共用主目录或各建一份，以 target membership 实际可行为准）。
- `project.yml` 声明 `knownRegions: [zh-Hans, en, ja, fr]`。

### 2. 系统级字符串

- 新建 `InfoPlist.xcstrings` 本地化：
  - 4 条权限描述（相机、麦克风、语音识别、相册）。
  - `CFBundleDisplayName`：中文"食材管家"，其余语言为 "Fresh Pantry"
    （拉丁/假名环境保留英文名，符合 iOS 常见做法；如需日文名后续可改）。
  - ShareExtension 的 Bundle 名称同步本地化。

### 3. 报错信息

- iOS 各 error enum（`AiError`、`BackupError`、`TextRecognizer` 错误等）的
  用户可见 message 改为本地化 key。
- API 错误响应体增加稳定 `code` 字段（如 `auth_missing`、`auth_expired`、
  `bad_request`、`quota_exhausted`、`misconfigured`），原 `message` 字段保留
  （日志与旧版客户端兜底）。iOS 按 `code` 映射本地化文案，未知 code 时回退
  通用错误文案。API 侧不做 Accept-Language 协商——翻译全部在客户端完成。
- 直接展示 `error.localizedDescription` 的 ~20 处：外层模板本地化
  （如 `error.photoReadFailed` = "读取照片失败：%@"），系统错误正文由
  iOS SDK 自身的本地化承担。

### 4. 通知

- `ExpiryScheduler` 的通知标题/正文改为 `String(localized:)`（含复数变体），
  按通知生成时的系统语言产出。

### 5. 食谱数据

实施调研修正（2026-07-05）：食谱浏览页运行时的数据源是 Supabase `recipes` 表
（`RemoteRecipeCatalog`），bundled `howtocook.json` 只是离线/首启种子，联网后
远程结果会写入磁盘缓存并成为主数据。因此译文不能做成"整份换语言的语料文件"
（联网刷新会被中文数据覆盖），改为**按 id 的翻译覆盖层（overlay）**：

- recipe-pipeline 新增 **translate 阶段**：
  - 输入中文语料，AI 翻译每道菜的 `name`、`description`、`ingredients[].name/
    unit/note`、`steps`、`tags` 到 en / ja / fr；`category`（10 个固定值）走
    固定映射表，保证全库译名一致。
  - 按 `recipe id + 可译字段内容 hash` 缓存（沿用 pipeline 现有 `.cache/`
    模式），重跑只增量翻译新增/变更的食谱。
- 产出 id 为键的覆盖层文件 `howtocook.i18n.en.json` / `.ja.json` / `.fr.json`，
  放入 `apps/ios/FreshPantry/Resources/` 打包。
- iOS 侧新增 `RecipeLocalizer`：按 app 实际界面语言（`Bundle.main.
  preferredLocalizations`）加载对应 overlay，套用到**任何来源**（bundle /
  磁盘缓存 / 远程）加载出的食谱上；中文界面不加载 overlay。overlay 中缺失的
  食谱（如远程新增、翻译失败的条目）保持中文原文。
- 用户自建食谱（Supabase `custom_recipes`）保持录入语言，不翻译。

### 6. 日期/数字格式化

- 移除 UI 展示层硬编码 locale（`RecipeDetailView` 的 `zh_CN` 星期格式等），
  改用 `Locale.current` / `Date.FormatStyle`。
- 数据序列化用的 `en_US_POSIX` ISO8601 formatter 保留（数据格式，非 UI）。

## 已知限制（本期不解决）

- **跨语言食材匹配**：食材↔食谱按名称匹配，同一语言环境内自洽；家庭成员
  用不同语言混录食材时，与食谱食材的匹配会失效。彻底解决需要 canonical
  ingredient ID，超出本期范围。
- App Store 元数据（商店页描述）在 App Store Connect 管理，不在本仓库范围。
- 用户自建食谱不做翻译。

## 错误处理

- 食谱翻译文件缺失/损坏 → 静默回退中文版，不打断使用（local-first calm）。
- API 返回未知 error code → 显示通用错误文案，不裸露原始 message。
- translate 阶段单条食谱翻译失败 → 记入 rejects，不阻塞整体构建，该条回退中文。

## 测试与验证

- 脚本检查 `Localizable.xcstrings` / `InfoPlist.xcstrings` 四语言无缺漏 key。
- pipeline translate 阶段单元测试（映射表、缓存命中、失败回退）。
- 现有 `npm run check`（api + recipe-pipeline）与 iOS 测试全部通过。
- 手工验证：切换系统语言到 en / ja / fr，走查主流程（首页、库存、食谱、
  购物、设置、通知、报错 toast）。
